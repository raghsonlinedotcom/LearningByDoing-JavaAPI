üìò Introduction to JPMS

JPMS (Java Platform Module System) was introduced in Java 9 to improve modularity, encapsulation, and maintainability of Java applications. It allows splitting large monolithic codebases into manageable modules.

‚ùì Why Use Modules?

  • Encapsulation of internal APIs
  • Faster startup and better performance
  • Explicit dependencies using requires
  • Clear separation of concerns
  • Enables strong encapsulation via exports

‚öôÔ∏è How to Create Modules

Each module requires a module-info.java descriptor:

module com.invoice.service {
  requires com.invoice.model;
  exports com.invoice.service;
}

Key keywords: requires, exports, opens, uses, provides ... with

üè¢ Real-World Examples (Accordion)

  • com.invoice.model ‚Äî POJOs, Domain Objects
  • com.invoice.service ‚Äî Business logic
  • com.invoice.api ‚Äî REST layer, public API
  • com.invoice.db ‚Äî Persistence layer
module com.invoice.db {
  requires java.sql;
  exports com.invoice.db;
}

Modules like com.hr.core, com.hr.auth, com.hr.payroll help encapsulate payroll and auth independently.

üíª CLI & Compilation

javac -d out --module-source-path src $(find src -name "*.java")
java --module-path out -m com.invoice.api/com.invoice.api.Main

‚ö†Ô∏è Common Pitfalls

  • Forgetting exports leads to IllegalAccessError
  • Mixing automatic modules (from JARs) and explicit modules can confuse the resolver
  • Third-party libs without module-info may need --add-exports or unnamed module hacks

‚úÖ Summary

  • Modules = clear contracts between packages
  • Boosts maintainability, readability, and security
  • Start small ‚Äî modularize new features first
