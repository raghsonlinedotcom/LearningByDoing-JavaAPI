📘 Module Fundamentals – Explained in Depth

🔹 What is a Module?

A Java module is a collection of code organized with boundaries. It’s a way to group related packages together with an explicit contract of what is public and what is private.

Real-time analogy:
  • 📦 Think of a module like a sealed shipping box (with a label).
  • ✅ It tells others what’s inside (exports).
  • 🔒 It hides what’s not meant to be seen.

🔧 Structure of a Module

A module is identified by its module-info.java file. This descriptor controls:

  • requires – dependencies
  • exports – public API packages
  • opens – reflection-only access
  • provides/uses – service providers

📂 Step-by-Step Example

Let’s say we are building a Billing System:

billing-system/
├── billing.core/           --> Invoice, TaxCalculator
├── billing.database/       --> InvoiceRepository
├── billing.api/            --> REST API (using billing.core)
├── billing.app/            --> Main entry point
billing.core/module-info.java
module billing.core {
  exports com.billing.core;
}
billing.database/module-info.java
module billing.database {
  requires billing.core;
  exports com.billing.database;
}
billing.api/module-info.java
module billing.api {
  requires billing.core;
  requires billing.database;
  exports com.billing.api;
}
billing.app/module-info.java
module billing.app {
  requires billing.api;
}

🚀 Benefits in This Setup

  • 🔐 Only classes explicitly exported are visible — true encapsulation.
  • 🧩 You know exactly what modules depend on each other (no classpath mystery).
  • 💥 If you remove a module, Java won't compile unless dependencies are declared.
  • 🔌 You can load services dynamically using provides and uses.

👨‍💻 Compile & Run

You compile the app using:

javac --module-source-path . -d out $(find . -name "*.java")

And run it using:

java --module-path out -m billing.app/com.billing.app.Main

📈 When to Use Modules

  • ✅ Large enterprise systems with dozens of packages
  • ✅ Microservices where strong boundaries matter
  • ✅ Libraries and frameworks to hide internal APIs
  • ✅ Security-sensitive applications that must limit reflective access

⚠️ When NOT to Use Modules

  • ❌ Small utility apps or CLI tools
  • ❌ Projects using Spring Boot (unless explicitly opened)
  • ❌ Libraries without proper modularization may break
