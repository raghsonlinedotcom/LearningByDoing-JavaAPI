Java Modules (JPMS) – Beginner to Advanced

A module is a self-contained unit of Java code with its own dependencies and visibility rules. Introduced in Java 9, it promotes strong encapsulation.

module com.myapp.billing {
  requires com.myapp.core;
  exports com.myapp.billing;
}

Only exported packages are accessible to other modules.

  • ❌ Classpath chaos
  • ❌ No encapsulation
  • ❌ Public = globally accessible
  • ✅ JPMS offers requires, exports, and opens to control this

ProblemJPMS Solution
No access controlexports only what’s needed

invoice-app/
├── invoice.core/
├── invoice.db/
├── invoice.service/
└── invoice.cli/

Each has its own module-info.java. Here’s one example:

module invoice.core {
  exports com.invoice.core;
}

javac --module-source-path . -d out $(find . -name "*.java")
java --module-path out -m invoice.cli/com.invoice.cli.App

module plugin.api {
  exports com.plugin.api;
}

module plugin.impl {
  requires plugin.api;
  provides com.plugin.api.TaskPlugin with com.plugin.impl.CleanupTask;
}

Used with:

ServiceLoader<TaskPlugin> loader = ServiceLoader.load(TaskPlugin.class);
loader.forEach(TaskPlugin::run);

  • ❌ Forgetting exports = access errors
  • ❌ Reflection fails without opens
  • ❌ No circular dependencies allowed

  • ✅ Enterprise-level architecture
  • ✅ Plugin systems and SPI
  • ❌ Avoid for small or Spring apps unless needed
